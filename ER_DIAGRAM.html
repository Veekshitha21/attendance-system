📋 Comprehensive Relationship Definitions

Primary Relationships with Cardinality and Constraints

1. STUDENTS → ATTENDANCE (One-to-Many)

Description: One student can have multiple attendance records across different classes and dates.

Cardinality: 1:M (One student to many attendance records)

Foreign Key: ATTENDANCE.studentId → STUDENTS.id

Referential Action: ON DELETE CASCADE (deleting a student removes all their attendance records)

Participation: Partial (a student may have zero or more attendance records)

Business Rule: Each attendance record must belong to exactly one student. A student can have attendance marked for multiple classes on multiple dates.

2. CLASSES → ATTENDANCE (One-to-Many)

Description: One class can have multiple attendance records for different students.

Cardinality: 1:M (One class to many attendance records)

Foreign Key: ATTENDANCE.classId → CLASSES.id

Referential Action: ON DELETE CASCADE (deleting a class removes all attendance records for that class)

Participation: Partial (a class may have zero or more attendance records)

Business Rule: Each attendance record must belong to exactly one class. A class can have attendance marked for multiple students on multiple dates.

Additional Constraint: UNIQUE(studentId, classId, date) ensures only one attendance record per student per class per date.

3. STUDENTS → LEAVES (One-to-Many)

Description: One student can submit multiple leave requests for different classes and date ranges.

Cardinality: 1:M (One student to many leave requests)

Foreign Key: LEAVES.studentId → STUDENTS.id

Referential Action: ON DELETE CASCADE (deleting a student removes all their leave requests)

Participation: Partial (a student may have zero or more leave requests)

Business Rule: Each leave request must be associated with exactly one student. A student can request multiple leaves for different classes and time periods.

4. CLASSES → LEAVES (One-to-Many)

Description: One class can have multiple leave requests from different students.

Cardinality: 1:M (One class to many leave requests)

Foreign Key: LEAVES.classId → CLASSES.id

Referential Action: ON DELETE CASCADE (deleting a class removes all leave requests for that class)

Participation: Partial (a class may have zero or more leave requests)

Business Rule: Each leave request must specify exactly one class. A class can receive leave requests from multiple students for various date ranges.

Relationship Attributes and Properties

STUDENTS-ATTENDANCE Relationship Attributes:
  • date: The specific date when attendance was recorded
  • status: Attendance status (Present/Absent/Late)
  • remarks: Optional notes or comments about the attendance
  • createdAt: Timestamp when the attendance record was created
  • Constraint: A student can have only one attendance record per class per date
STUDENTS-LEAVES Relationship Attributes:
  • startDate: The beginning date of the leave period
  • endDate: The ending date of the leave period
  • reason: The reason for requesting leave (max 500 characters)
  • status: Leave approval status (Pending/Approved/Rejected)
  • createdAt: Timestamp when the leave request was submitted

Cardinality Notation Guide

1:M (One-to-Many):

One entity instance on the "one" side can be associated with multiple entity instances on the "many" side. However, each entity instance on the "many" side can be associated with only one entity instance on the "one" side.

Example: One STUDENT (1) can have many ATTENDANCE records (M), but each ATTENDANCE record belongs to only one STUDENT.

Participation Constraints:
  • Total Participation: Every entity instance must participate in the relationship (currently not enforced in this system)
  • Partial Participation: Entity instances may or may not participate in the relationship (used in all relationships in this system)
Referential Integrity Actions:
  • CASCADE: When a parent record is deleted, all related child records are automatically deleted
  • Example: Deleting a student automatically deletes all their attendance records and leave requests
  • Benefit: Maintains database consistency and prevents orphaned records

🔍 Legend & Reference

PK - Primary Key

Uniquely identifies each record in the table. No two records can have the same primary key value.

FK - Foreign Key

References the primary key of another table, establishing relationships between tables.

UQ - Unique Constraint

Ensures all values in the column are different. No duplicates are allowed.

1:M - One-to-Many

One record in the parent table relates to multiple records in the child table.

CASCADE Delete

When a parent record is deleted, all related child records are automatically deleted.

Timestamp Fields

createdAt and updatedAt are automatically managed by the database for audit trails.

🔒 Data Constraints & Integrity

Constraints Applied

Students Table

• rollNumber: UNIQUE - Each student has a unique registration number

• email: UNIQUE - Each student email must be unique

• status: DEFAULT 'Active' - New students are active by default

Classes Table

• code: UNIQUE - Each class has a unique code (CS301, DBMS, CN, etc.)

• name: NOT NULL - Class name is required

Attendance Table

• UNIQUE(studentId, classId, date) - Only one attendance record per student per class per date

• Cascade Delete - Deleting student removes all attendance records

• Foreign Key constraints - Prevents invalid references

Leaves Table

• status: DEFAULT 'Pending' - New leave requests start as pending

• Cascade Delete - Deleting student removes all leave requests

• startDate < endDate - Start date must be before end date (validated by application)
